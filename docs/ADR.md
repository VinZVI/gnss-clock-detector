## ADR-001: PostgreSQL для Production, SQLite для Dev
**Статус:** Принято | 2026-03 (Обновлено 2026-04)

**Контекст:** SQLite не подходит для облачных сред (Render.com) из-за эфемерной файловой системы и отсутствия параллельной записи.

**Решение:** 
- **Production:** PostgreSQL (Managed service на Render). 
- **Development:** SQLite (локальный файл `db.sqlite3`).

**Последствия:** Требуется настройка `DATABASE_URL` в окружении. Приложение автоматически переключается между диалектами через SQLAlchemy.

## ADR-002: Async ETL через threading + polling
**Статус:** Принято | 2026-04

**Контекст:** Gunicorn worker timeout 120s; ETL занимает 2-5 минут (скачивание файлов NASA).

**Решение:** POST /api/admin/etl → daemon-поток → 202 Accepted. Polling через GET /api/admin/etl/status/<job_id>.

**Последствия:** Состояние теряется при рестарте. Приемлемо при текущей нагрузке.

## ADR-003: IGS3 long-name format для NASA
**Статус:** Принято | 2026-04

**Контекст:** NASA переименовал файлы с GPS-недели 2238 (ноябрь 2022). Старый `igu*.sp3.Z` не существует.

**Решение:** Формат `IGS0OPSULT_<YYYYDDD><HH>00_02D_30S_CLK.CLK.gz`. Пробуем 8 анализ-центров: COD, EMR, GFZ, TUG, SHA, JPL, MIT, ESA.

**Последствия:** Гарантий нет, что конкретный центр опубликовал файл для данного слота.

## ADR-004: Bearer Token для NASA CDDIS
**Статус:** Принято | 2026-04

**Контекст:** ConnectionResetError 10054 на Windows при OAuth redirect с Basic Auth.

**Решение:** EDL API `/api/users/find_or_create_token`, кеш в `.token_cache` (60 дней).

**Последствия:** Нужен мониторинг истечения токена. `.token_cache` в `.gitignore`.

## ADR-005: Rolling window MAD вместо глобального
**Статус:** Принято | 2026-04

**Контекст:** Глобальный MAD некорректен для нестационарных часовых рядов (дрейф, скачки).

**Решение:** Скользящее окно `window_size=15` точек вокруг каждой. Параметр через API.

**Последствия:** Граничные точки имеют меньше соседей (обрабатывается корректно через `max(0, i-half)`).

## ADR-006: YYYYDDD в именах каталогов FTP
**Статус:** Принято | 2026-03

**Контекст:** FTP GLONASS-IAC использует YYYYDDD (год 2-цифры + день года 3-цифры).

**Решение:** `date_to_dir()` в gps_time.py: `f"{yy:02d}{doy:03d}"`. Пример: 2026-03-20 → "26079".

**Последствия:** В 2100 году потребуется смена формата (неактуально).

## ADR-007: Абсолютные пути к SQLite DB
**Статус:** Принято | 2026-04

**Контекст:** Разные точки входа (app, etl, migrate) имеют разные `instance_path`, что приводило к созданию дубликатов `db.sqlite3` в разных папках при использовании относительных путей в .env.

**Решение:** `config.py` принудительно резолвит `GNSS_DB_PATH` в абсолютный путь относительно корня проекта.

**Последствия:** Исключена десинхронизация данных между веб-интерфейсом и ETL-скриптами.

## ADR-008: Гибридная маршрутизация (SPA + MPA)
**Статус:** Принято | 2026-04

**Контекст:** Пользователи жаловались, что данные спутника (графики, аналитика) отображались только после Refresh. Это происходило из-за того, что SPA-переход в `index.html` показывал неполную "заглушку" вместо загрузки полноценной страницы `satellite.html`.

**Решение:** 
- Навигация к спутникам переведена на `window.location.href` (полная перезагрузка).
- В `app.py` добавлены явные маршруты для всех страниц (включая `/admin/etl`), чтобы Refresh работал корректно без 404.

**Последствия:** Повышение надежности UX при сохранении SPA-функциональности для панели управления ETL.

## ADR-009: Расширение метаданных (Номер аппарата)
**Статус:** Принято | 2026-04

**Контекст:** Для идентификации конкретных спутников в группировке ГЛОНАСС недостаточно системного номера (R01, R02), так как аппараты могут меняться. Требуется заводской номер (например, 760).

**Решение:** 
- Добавлена колонка `sat_num` в таблицу `satellite_meta`.
- Парсер `.glo` обновлен для извлечения 3-й колонки файла.
- Добавлены автоматические миграции в `app.py` и `etl.py` для расширения существующей схемы БД.

**Последствия:** Упрощение сопоставления аномалий с конкретными физическими аппаратами.
